# 鼠标指针配置管理器 Bug 修复执行方案

本文档用于把当前 bug 合辑整理成可执行的开发任务。原则是：一次只修一个明确问题，每完成一步就提交一次；先补保护和验证，再改交互和打包，避免旧问题未确认就引入新 bug。

## 当前问题总结

这些问题可以归成 7 类：

1. 导入与匹配不稳定：`rar` 支持不完整；部分 `.ani` 无缩略图；手写、精确选择、位置选择、个人选择偶尔匹配错；资源包拖拽入口和普通文件拖拽入口行为不统一。
2. 鼠标方案编辑交互弱：替换鼠标指针盒子不能直接拖文件；方案 list 图标缺少悬停反馈、删除/移回入口和动静标识；用户不容易发现图标可点、可聚焦。
3. 方案切换状态不清晰：切换后当前方案没有实时刷新；下次切换不显示或不包含计时切换；计时切换 UI 不够直观，顺序/随机逻辑和用户预期不一致。
4. 计时切换需要重做：时间组件宽度应自适应；“每 X 秒/分钟”组件应紧凑；需要 4x4 宫格、多选、全选、拖拽排序；顺序模式按多选排序切换，随机模式在多选范围内随机。
5. 打包和发布链路不稳定：已打包软件内找不到包含 PyInstaller 的 Python；生成安装包偶尔显示黑框；Release 更新后 README 和网页下载链接没有同步；x64/arm64 架构策略未定。
6. 启动和后台驻留需要确认：打开程序等待时间过长；自启动是否真的启动缺少可靠检查；关闭后托盘图标或后台状态是否保留需要真实验证。
7. 在线资源库信息过期：资源库地址需要改为 `http://8.135.33.2:5002/`，并在入口处添加图标。

## 开发总原则

- 先创建分支：`git switch -c codex/bugfix-stability-plan`。
- 每一步只解决一个主题，提交前看 `git diff`，不要顺手重构无关代码。
- 每步提交格式建议：`fix: ...`、`feat: ...`、`docs: ...`、`test: ...`。
- 不要把 UI 重做、导入兼容、打包运行时一次塞进同一个提交。
- 修改前保留一组真实素材：`.cur`、普通 `.ani`、无法预览的纯 `.ani`、`.zip`、`.rar`、`.7z`、一个包含多份 `.inf` 的压缩包、一个手写/精确/位置/个人选择命名不标准的主题包。
- 任何涉及系统鼠标、注册表、自启动、托盘的修改，都要在 Win10 和 Win11 真机验证。

## 分步修复路线

### 1. 建立回归保护和真实素材清单

目标：先让程序员知道每步改完怎么验证，避免越修越乱。

要做：

- 新增或更新发布测试清单，把本次所有问题转成可勾选项。
- 新建 `docs/test-assets.md`，记录每类测试素材的来源、文件名、覆盖问题、预期结果。
- 给导入、角色匹配、切换、打包、自启动各列一条最小手测路径。

验收：

- 程序员能按清单复现每个问题。
- 没有真实素材的问题标记为“需要补素材”，不要假装已修复。

提交：

```text
docs: add bug regression checklist
```

注意：

- 这一步不改代码。
- 当前仓库 `README.md`、`README.en.md`、`介绍.md`、网页和 Release 链接存在同步风险，后面统一处理。

### 2. 修复鼠标角色匹配错误

目标：手写、精确选择、位置选择、个人选择不再被错误当成替代/普通鼠标指针。

要做：

- 扩展 `map_files_to_roles` 的规则，补齐 `Pin`、`Person`，并重新检查 `NWPen`、`Crosshair`、`Hand` 的关键词。
- 修正 `parse_inf_mapping` 的 `alias_to_reg`。当前 `hand` 映射到 `NWPen` 风险很高，应该区分 `pen/nwpen` 和 `hand/link`。
- `number_roles` 目前只覆盖 15 个角色，需要覆盖完整 17 个 `CURSOR_ROLES`，包含 `Pin` 和 `Person`。
- 对同一个文件不能被多个角色占用的逻辑要保留，但要先按 INF 精确映射，再按关键词，最后按编号兜底。

验收：

- 包含 `pen`、`cross`、`pin`、`person`、`hand/link` 的主题包导入后，角色落位正确。
- 没有 `Pin`、`Person` 的老主题包仍可导入，不报错。
- 导入后应用系统鼠标方案正常。

提交：

```text
fix: improve cursor role matching
```

注意：

- 这一步只改匹配，不改 UI。
- 不要为了匹配文件名加入过宽关键词，例如单独用 `p`、`select` 之类很容易误伤。

### 3. 修复 RAR 导入兼容

目标：`.rar` 添加后不再直接显示错误；失败时提示原因可理解。

要做：

- 明确采用一种 RAR 解压策略：优先推荐捆绑 `7z.exe/7z.dll` 到 `runtime/7zip`，其次再使用 `rarfile`。
- `extract_import_package` 中 `.rar` 分支不要吞掉异常后直接走不确定的 `tar -xf`。应记录具体失败原因，并按顺序尝试可用后端。
- 打包 spec 要把 `runtime/7zip` 包进绿色版和完整版。
- 错误提示要区分“文件损坏”“不支持加密 RAR”“缺少解压运行时”。

验收：

- 普通 RAR 可导入。
- 加密 RAR 不闪退，提示“暂不支持加密压缩包”。
- 没有安装 WinRAR 的干净系统也可导入普通 RAR。

提交：

```text
fix: support rar import with bundled extractor
```

注意：

- `rarfile` 本身通常依赖外部解压程序，不能只因为 `requirements.txt` 里有 `rarfile` 就认为绿色版已经支持 RAR。
- 先确认 7-Zip 许可和分发方式，再放进 release。

### 4. 修复 `.ani` 缩略图和预览兜底

目标：纯 `.ani` 即使不能生成动画预览，也要有稳定缩略图或占位图，不影响应用。

要做：

- 梳理 `ani_frame_paths`、`render_cursor_with_windows` 和 `cursor_preview_image_sized` 的失败路径。
- 对无法拆帧的 `.ani` 使用 Windows `LoadImageW + DrawIconEx` 取首帧。
- 如果 Windows API 也失败，显示统一占位图：文件类型、文件名、`动` 标识，而不是空白。
- 资源库预览、方案行预览、右侧实时预览共用同一套兜底函数。

验收：

- 普通 `.ani` 能动。
- 无法拆帧的 `.ani` 至少显示首帧或占位图。
- 点击应用后仍可正常写入系统鼠标。

提交：

```text
fix: add ani preview fallback
```

注意：

- 不要因为预览失败就阻止应用，预览能力和系统应用能力要分开。
- 预览失败要写入错误记录，但 UI 不应频繁弹窗打断用户。

### 5. 增加素材动静标识

目标：替代鼠标指针盒子中的素材左上角显示圆形文字图标：`.ani` 显示“动”，`.cur` 显示“静”，其他格式暂不匹配。

本轮修订：标识不能遮挡鼠标图标，因此不再由通用预览控件直接覆盖绘制。

要做：

- 在方案行、资源宫格和替换素材区域统一使用文件类型分类函数。
- 只对 `.ani`、`.cur` 显示标识。
- 方案编辑页把标识放在角色名称旁；替换资源池把标识放在图标下方；资源库只在方案名称旁显示汇总。
- 资源库预览小图必须调用无标识预览路径，避免每个小图重复显示。

验收：

- `.ani` 显示“动”，`.cur` 显示“静”。
- 图片格式、未知格式不显示动静标识。
- 缩放窗口时标识不遮住图标、删除按钮和文字。
- 资源库方案名旁显示 `动 N` / `静 N` 汇总，预览小图不显示逐个标识。

提交：

```text
feat: show cursor type badges
```

注意：

- 这一步只做标识，不顺带修改拖拽或删除逻辑。

### 6. 替换鼠标指针盒子支持直接拖入文件

目标：用户能把 `.cur`、`.ani`、图片、压缩包直接拖到替换区域；行为和点击导入一致。

要做：

- 明确两个拖拽入口：单个角色盒子拖入文件是“替换该角色”；空白替换盒子或资源导入区拖入压缩包是“导入新方案”。
- 当前 `_v3_add_row` 已给行内控件注册 DND；需要检查“替换鼠标指针盒子”实际对应哪个 Widget，并补齐 `drop_target_register`。
- 拖入多个文件时按顺序处理，单个角色只取第一个有效文件；资源导入区可批量处理多个压缩包。
- 拖入错误类型时只在状态栏提示，不要崩溃。

验收：

- 拖 `.cur/.ani` 到指定角色，路径和预览立即更新。
- 拖图片到指定角色，可以生成预览，应用时可转换为 `.cur`。
- 拖压缩包到资源区，导入为新方案。

提交：

```text
feat: support drag and drop for replacement box
```

注意：

- DND 要兼容中文路径和带空格路径。
- 拖入压缩包到角色行时不要错误当作单个鼠标文件。

### 7. 方案 list 图标悬停、移回和轻微晃动

目标：鼠标滑到方案 list 对应图标时，图标显示虚线焦点框，右上角出现小 X；点击 X 后对应鼠标方案移动回替换鼠标指针盒子；悬停时图标轻微晃动。

要做：

- 先定义“移动回替换鼠标指针盒子”的数据含义：建议把已选角色从 `self.selected` 移除，并把该素材放回可选素材池或替换盒子候选列表。
- 在图标组件上增加 hover 状态：虚线边框、小 X、鼠标指针样式。
- 小 X 只在 hover 时显示，点击后调用单一方法，例如 `unassign_role_to_pool(role.reg_name)`。
- 晃动动画要轻，建议 2 到 3 像素横向偏移，150 到 250ms 内停止；不要持续循环占用 CPU。

验收：

- 悬停显示虚线框和小 X。
- 点击小 X 只移除当前角色，不影响其他角色。
- 鼠标离开后样式恢复。
- 快速滑过大量图标不会造成 UI 卡顿。

提交：

```text
feat: add hover actions for scheme icons
```

注意：

- Tk/ttk 对虚线边框和覆盖按钮不如 Qt 灵活，必要时用 `Canvas` 包一层图标单元。
- 不要把动画绑定到资源库所有动态图帧计时器上，避免和 `.ani` 预览互相影响。

### 8. 在线资源库地址和图标

目标：在线资源库入口改为 `http://8.135.33.2:5002/`，并添加图标。

要做：

- 更新 `RESOURCE_URL`。
- 资源库按钮使用现有图标系统，或在 `assets` 中添加一个资源库图标。
- 同步更新 README、英文 README、介绍页、网页下载/资源入口说明。

验收：

- 点击“打开资源库网页”打开新地址。
- Edge app 模式和默认浏览器兜底都可用。
- 文档和网页没有旧资源库地址。

提交：

```text
feat: update online resource library entry
```

注意：

- HTTP 地址可能被安全软件或浏览器策略拦截，失败时要提示用户复制链接到浏览器。

### 8.1 资源库选择和统一操作

目标：资源库按成熟桌面应用方式管理方案，不在每个卡片上重复堆叠应用和删除按钮。

要做：

- 列表视图和宫格视图都支持点击卡片选中。
- 右下角只保留一个全局 `应用` 和一个全局 `删除`。
- `应用` 和 `删除` 都只针对当前单选方案。
- 切换列表/宫格、刷新资源库后保留仍然存在的单选项，清理已经不存在的选中项。
- 资源库卡片只负责展示名称、汇总标识和预览，不再承担独立动作按钮。

验收：

- 选中 0 个方案时，应用和删除不可用。
- 选中 1 个方案时，应用和删除可用。
- 点击另一个方案后，旧方案自动取消选中，新方案成为唯一选中项。
- 删除后方案从资源库和方案下拉框消失，并提示回收区位置。
- 列表视图和宫格视图的选择状态一致。

### 9. 重做计时切换 UI 和数据模型

目标：计时切换具备紧凑时间组件、4x4 方案宫格、多选、全选、拖动排序、顺序/随机模式。

要做：

- 数据模型从单个 `scheme` 扩展为 `selected_schemes: list[str]`、`order: "顺序" | "随机"`、`interval_seconds`、`sequential_index`。
- UI 表达为：`每 [数字输入] [秒/分钟下拉]`，组件宽度按内容自适应，不铺满整行。
- 方案选择改成 4x4 宫格，每格显示“正常选择”图标和方案名。
- 添加“全选”预选项，点击方案格切换选中状态。
- 选中列表左侧提供拖动排序手柄，顺序模式按排序切换，随机模式只在选中集合内随机。

验收：

- 只选 3 个方案时，顺序切换只在这 3 个中循环。
- 拖动排序后，下一轮顺序按新排序执行。
- 随机模式不会选到未勾选方案。
- 重启程序后计时配置完整恢复。

提交：

```text
feat: rebuild timer switch selection
```

注意：

- 兼容旧配置：如果旧配置只有 `scheme: "顺序"` 或 `scheme: "__random__"`，首次读取时迁移为全选。
- 这一步会影响 `pick_scheduled_scheme`、`run_background`、`run_pystray_tray` 和 UI 保存逻辑，必须重点回归后台切换。

### 10. 修复当前方案和下次切换实时显示

目标：方案切换后 UI、托盘菜单、状态栏能实时显示当前方案；下次切换包含时间、星期和计时切换。

要做：

- 所有应用入口统一走一个 `apply_scheme_and_update_state(theme)` 函数，内部调用 `apply_library_scheme` 后更新 `current_scheme`。
- `next_switch_text` 增加计时切换计算，显示剩余时间或下一次触发时间。
- 后台和托盘中的 `last_timer_at`、`timer_index` 要持久化或写入状态文件，否则重启后“下一次切换”无法准确显示。
- UI 保存切换设置后立即刷新当前方案、下次切换文案和托盘菜单。

验收：

- 手动应用、资源库应用、时间切换、星期切换、计时切换都会更新“当前配置”。
- 托盘右键菜单能显示当前配置和下次切换。
- 计时切换启用后能看到下一次切换，不再为空。

提交：

```text
fix: refresh scheme switch status in real time
```

注意：

- 不要只更新 UI 文案；必须确保设置文件里的 `current_scheme` 和实际系统应用成功保持一致。
- 如果应用失败，不要提前写入当前方案。

### 11. 安装包生成内置依赖并消除黑框

目标：打包后的软件可以生成安装包，不再提示找不到包含 PyInstaller 的 Python；生成过程不显示黑框。

要做：

- 决定运行时方案：推荐在绿色版中捆绑专用 `runtime/python` 和 PyInstaller，`find_python_with_pyinstaller` 优先查 `resource_path("runtime/python/python.exe")`。
- spec 文件把 `runtime` 目录纳入 `datas`，并确保 release 资产使用完整版 spec。
- 所有 `subprocess.run/Popen` 打包相关调用都带 `CREATE_NO_WINDOW`、`stdout/stderr` 捕获或重定向。
- 如果运行时不存在，错误提示改成“当前版本未包含安装包生成运行时，请下载完整版”，不要让用户去装 `requirements.txt`。

验收：

- 在未安装 Python 的干净 Windows 上，绿色版可以生成安装包，或清晰提示需要完整版。
- 生成安装包期间无黑色控制台窗口。
- 打包失败时能打开 `pyinstaller_error.log` 查看原因。

提交：

```text
fix: bundle installer build runtime
```

注意：

- 这一步可能显著增大 exe 体积，需要在 Release 说明里讲清楚。
- PyInstaller 在冻结程序内再次调用 PyInstaller 是高风险点，必须在干净系统测试，不要只在开发机验证。

### 12. 明确 x64 和 arm64 发布策略

目标：不要盲目增加架构，先减少发布复杂度。

建议：

- 先只发布 Windows x64，这是大多数用户需要的稳定版本。
- arm64 可以作为后续独立目标，只有在有真实 Windows ARM 用户、可用 ARM64 构建机或虚拟机、并能完成手测时再加。
- PyInstaller 通常不能在 x64 机器上直接可靠交叉编译 arm64 Windows exe，最好在对应架构机器上构建。

验收：

- README 和 Release 说明明确当前支持架构。
- 下载按钮不会让用户误以为有 arm64 版本。

提交：

```text
docs: document release architecture support
```

注意：

- 架构不是越多越好。没有测试能力的架构会扩大 bug 面。

### 13. Release 后同步 README 和网页下载链接

目标：更新 Release 后，README 和网页自动或半自动同步到对应 exe。

要做：

- 统一下载资产名：`MousePointer_Portable.exe` 和需要时的 `MousePointer_Installer.exe`。
- README 下载链接改成 GitHub latest release 链接，减少硬编码版本号。
- `docs/index.html` 和 `docs/script.js` 的下载按钮同步使用同一来源。
- 在 `.github/workflows/release.yml` 中增加发布说明，列明资产和哈希。

验收：

- 打 tag 发布后，README、网页下载按钮都能下载对应 exe。
- 页面不再指向旧版本 `v2.0.0`。


注意：

- GitHub Release 中文资产名此前有规范化问题，建议继续用英文资产名。

### 14. 优化启动速度

目标：打开程序等待时间明显降低。

要做：

- 启动时先显示主窗口，再异步做资源扫描、默认方案导入、预览缓存生成。
- `ensure_default_schemes` 不要每次启动都同步扫描/解压所有内置包。
- 资源库预览只在进入资源库页时生成，且限制首屏数量；滚动到可见区域再加载后续预览。
- 大型图片和 `.ani` 帧预览加缓存，避免重复渲染。

验收：

- 冷启动到窗口可见时间有明确记录。
- 进入资源库页时可以显示加载状态，不阻塞整个程序。
- 首次启动和二次启动都可接受。

提交：

```text
perf: reduce startup blocking work
```

注意：

- 不要把耗时工作丢到后台后忘记错误处理。后台导入失败仍要写错误记录并给状态提示。

### 15. 检查自启动和退出后托盘保留

目标：确认自启动真的启动，退出后托盘或后台状态符合设置。

要做：

- 诊断信息中显示 Run 项、启动文件夹快捷方式、任务计划命令、托盘 PID、后台 PID、实际进程路径。
- “立即测试后台启动”根据当前“隐藏任务栏图标”设置启动 `--tray` 或 `--background`，而不是固定后台。
- 关闭窗口时，若选择保留托盘，则托盘图标必须出现；若隐藏任务栏图标，则不显示托盘但后台进程必须存在。
- 清理 stale pid，避免旧 pid 文件导致误判。

验收：

- Win10/Win11 重启后，自启动状态和设置页显示一致。
- 关闭窗口后托盘图标保留；右键退出后进程结束。
- 隐藏任务栏图标开启后，托盘不显示但后台切换仍工作。

提交：

```text
fix: verify autostart tray and background modes
```

注意：

- 这一步必须真机重启测试，单靠代码阅读不够。
- Windows 安全中心或杀毒软件可能拦截启动项，诊断信息要能帮助定位。

## 建议的最终验收顺序

1. 导入测试：`.cur`、`.ani`、无法预览 `.ani`、`.zip`、`.rar`、`.7z`、`.exe`、文件夹、多 INF 压缩包。
2. 匹配测试：正常选择、手写、精确选择、位置选择、个人选择、链接选择全部落位正确。
3. 编辑交互测试：拖拽替换、悬停虚线框、小 X 移回、轻微晃动、动静标识。
4. 切换测试：时间切换、星期切换、计时顺序、计时随机、当前方案、下次切换、托盘菜单。
5. 打包测试：开发机、干净 Win10、干净 Win11；无 Python 环境下生成安装包。
6. 发布测试：Release 资产、README 下载、网页下载、自动更新、哈希一致。
7. 后台测试：关闭窗口保留托盘、右键退出、开机自启动、隐藏任务栏后台。

## 不建议做的事

- 不要一次把所有 UI 改成新框架。当前 `main.py` 已经有多轮 monkey patch，继续大改会扩大风险。
- 不要把预览失败当成应用失败。`.ani` 预览和 Windows 鼠标应用是两条链路。
- 不要只在开发机测试安装包生成。开发机有 `.venv`，无法暴露“打包后找不到 PyInstaller”的真实问题。
- 不要新增 arm64 Release，除非有真实 arm64 构建和测试环境。
- 不要让后台轮询频率无限变快。输入法切换可以快，普通计时和星期切换不需要高频占用 CPU。

## 推荐优先级

第一优先级：角色匹配、RAR 导入、`.ani` 预览兜底、安装包生成运行时、自启动和托盘状态。这些最影响“能不能用”。

第二优先级：计时切换重做、当前方案/下次切换实时刷新。这些影响核心体验，但改动面更大，需要在基础稳定后做。

第三优先级：悬停小 X、晃动、动静标识、资源库图标和网页同步。这些能明显提升体验，但不应压过稳定性修复。
