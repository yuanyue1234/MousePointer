# Python 架构整理计划

## 原则

先修稳定性，再拆文件。拆分时保持现有 Python 入口、设置文件和资源目录兼容。

## 当前问题

- `main.py` 同时承担启动、导入、预览、应用、后台、自启动、打包等职责，文件过大。
- `fluent_ui.py` 同时包含通用组件、方案页、资源库、切换页和设置页，UI 维护成本高。
- 部分纯逻辑没有独立测试，导致重构风险偏高。

## 拆分顺序

1. 抽出纯逻辑：角色匹配、INF 解析、文件类型识别、资源摘要。
2. 抽出系统能力：应用鼠标方案、备份恢复、自启动、托盘和后台。
3. 抽出导入能力：压缩包、RAR、7z、安装器、单文件和多 INF。
4. 抽出 UI 页面：方案编辑页、资源库、切换页、设置页。
5. 保留 `main.py` 作为兼容入口，直到发布包验证通过。

## 测试要求

- 每次抽模块前先补对应单元测试。
- 系统鼠标、注册表、自启动、托盘只做真机验收，不用伪造测试替代。
- 拆分后的行为必须与拆分前一致，除非对应 bug 已在计划中明确修复。
